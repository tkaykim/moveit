# 예약·수강권·멤버십·주문 도메인

학원마다 규칙이 다르다는 것이 요구사항 자체다. 아래는 **메커니즘**이고, 값은 전부 학원별 데이터다.

## 수업 위계

`classes`(마스터) → `recurring_schedules`(주간 패턴) → `schedules`(날짜별 인스턴스).

수업은 **수업군**(`class_groups`)에 속한다. 수업군이 커버리지·할인·특별수업 여부를 결정한다.
`class_groups.is_special = true` 인 수업군 = **특별수업** — 무제한 패스로도 못 듣고 별도 결제가 필요하며, 멤버십은 여기에 *할인*으로 혜택을 준다. (워크샵·단발 특강류)

`classes.class_group_id IS NULL` = **예약 불가**. 단 이 규칙은 그 학원이 수업군을 하나라도 만든 뒤에 발효된다(레거시 학원은 기존 경로 유지). 외부 일정 도구가 만든 새 수업이 태깅 없이 결제 대상이 되는 사고를 막는 안전선이다.

`classes.audience_membership_id` = 특정 멤버십 전용 수업. **UI가 아니라 RLS에서 가려진다** — 비회원의 REST 응답 자체에 없어야 한다.

## 수강권

`tickets`(상품) → `user_tickets`(보유).

- `ticket_type`: `COUNT`(횟수권) / `PERIOD`(기간권)
- `is_fixed_weekly`: 주 N회 고정반 상품 — 구매 시 학생이 반 하나를 고르고 `user_tickets.fixed_class_id`에 박힌다
- `start_mode`: `IMMEDIATE` / `FIRST_BOOKING` — 후자는 발급 시 시작·만료가 NULL이고 **첫 예약이 확정될 때 그 수업의 KST 날짜로 개시**된다. 그 예약을 취소해도 되돌아가지 않는다.
- `valid_days`(일) 또는 `valid_months`(달력 개월)

**커버리지 판정 순서**: ① `ticket_classes`(수업 개별 매핑)가 있으면 그것 ② 없으면 `ticket_coverage`(수업군, `is_active=true`) ③ 둘 다 없으면 레거시 `is_general` 동작.

**차감할 수강권 선택 순서**: ① 같은 주문에서 산 것으로 지정된 수강권(`order_items.source_purchase_item_id`) ② 커버하는 기간권(무제한이 커버하는데 횟수를 태우지 않는다) ③ 커버하는 횟수권 중 만료 임박 순 ④ 고정반 수강권은 그 반에서만.

⚠ **만료일 계산에 두 계보가 있다.** 기존 데이터는 `start + valid_days`(exclusive)로 저장돼 있고, 신규 발급 규칙은 포함일수(`start + valid_days - 1`)다. `lib/date/kst.ts`가 둘을 분리해 갖고 있으며, 기존 수강권의 만료 의미는 바뀌지 않았다. 규칙 통일은 제품 결정이 필요한 미해결 항목이다.

## 예약

**확정 트랜잭션 검증 순서(고정)**: 일정 행 잠금 → ① 수업군 존재 → ② 특별수업 규칙 → ③ audience 자격 → ④ 예약 창구 → ⑤ 커버리지 → ⑥ **수업일 기준** 유효기간 → ⑦ 중복 예약 → ⑧ 정원 → 차감+생성.

정원은 `CONFIRMED` + `COMPLETED` + **유효한 홀드(`PENDING` AND `hold_expires_at > now()`)**를 센다.

**예약 창구**(`booking_policy` jsonb, 학원 기본 → 수업 오버라이드, 필드 단위 병합):
```json
{ "open": {"daysBefore":1,"time":"17:00"}, "close": {"minutesBefore":0}, "cancelUntil": {"minutesBefore":0} }
```
`open` 없음 = 상시 예약 가능.

**취소·복구는 서버가 판정한다.** 클라이언트가 "복구할지"를 정하지 않는다. 반복 취소 요청에도 복구는 정확히 한 번.

**고정반 자동 예약**은 예약 창구를 면제받는다(고정 등록생이므로). 회차가 아직 없거나 정원이 찼으면 **건너뛰고 잔여 횟수를 보존**하며, 실패분은 운영자 큐로 간다. 회차 배정 실패가 주문 확정을 막지 않는다.
⚠ 자동 예약은 **고정반 상품에만** 적용된다. 무제한 패스에 적용되면 30일치 전 수업이 자동 예약되는 사고가 난다 — 그래서 조건이 봉인돼 있고 회귀 테스트가 지키고 있다.

## 멤버십

학원별로 N개 정의한다(전문반/입시반/…). 학생당 학원별 **활성 1개**(부분 유니크 인덱스).

혜택 4종 조합: ① 수강권 번들(가입 시 자동 발급) ② 전용수업 접근 자격 ③ **할인**(수업군 또는 특정 상품에 %) ④ 표기 혜택(연습실·사물함 등, 시스템 밖).

- 부여와 번들 발급은 **하나의 트랜잭션**이다. 한쪽만 남는 상태를 허용하지 않는다.
- `SUSPENDED`는 할인도 자격도 주지 않는다. 재개는 다른 활성 멤버십이 없을 때만.
- **만료돼도 번들 수강권은 회수하지 않는다** — 이미 제공된 계약상 혜택이며 자기 만료일까지 산다. 만료자의 미래 전용수업 예약은 자동 취소하지 않고 **운영 큐**로 보낸다.
- 할인은 **최대 1개**(가장 큰 %)만 적용되고, 레거시 `discounts`와 중첩하지 않는다. 적용 근거(멤버십·%)는 주문 항목에 스냅샷된다.
- `visibility='hidden'`이면 학생에게 존재 자체가 보이지 않는다(상담제 학원 대응).

## 주문·결제

`order_groups` + `order_items` — 여러 수업과 수강권 구매를 한 주문으로 묶는다.

**상태**: `DRAFT → PENDING_PAYMENT → PAYMENT_APPROVED → CONFIRMED`, 그리고 `→ CANCELED` / `→ EXPIRED`(계좌이체 24시간) / `PAYMENT_APPROVED → FULFILLMENT_FAILED →(멱등 재시도)→ CONFIRMED`.

- **장바구니는 결제 전에 거른다.** 사전 검증이 항목별 사유를 돌려주고, 학생이 불가 항목을 빼고 진행한다. "결제 후 취소"는 하지 않는다.
- **가격은 서버가 다시 계산한다.** 클라이언트가 보낸 금액은 신뢰하지 않는다.
- 주문 생성 시 상품명·타입·횟수·유효기간·개시방식·금액을 **스냅샷**한다. 계좌이체가 24시간 대기하는 동안 상품이 수정돼도 산 대로 받는다.
- 계좌이체 주문은 **PENDING 예약으로 좌석을 홀드**하고 24시간 뒤 만료 스윕이 자리를 푼다. 입금확인과 만료는 같은 행 잠금으로 정확히 하나만 이긴다.
- 이행은 `finalize_order_group` 하나로 통일된다(계좌이체·카드·현장결제 동일). 이미 확정된 주문을 다시 확정해도 중복 발급되지 않는다.
- 비로그인 상태에서 수강권 구매는 **거부**한다(발급 대상이 없어 이행 불가한 주문이 생기던 구멍을 막았다).

## 환불

법적 위계: **① 강행 법령·등록 업종 기준 → ② 학원의 유효한 고지·개별 약정 → ③ 학원 설정 프리셋 → ④ 스태프 조정(감사 기록)**.

프리셋 4종: `ACADEMY_STATUTORY`(학원업 — 교습기간 경과 비율 기준) / `SPORTS_FACILITY_CONTINUOUS`(체육시설업 — 이용분 + 총액 10% 상한 위약금) / `CUSTOM_STEP` / `NON_REFUNDABLE_WHERE_LAWFUL`.
공통: **이용분은 실제 결제액 기준**(정가 아님), 카드수수료를 소비자에게 따로 전가하지 않는다.

⚠ 프리셋은 **옵트인**이다. 기존 수강권에 자동 적용되지 않으므로 기존 환불 결과는 바뀌지 않았다. 기존 상품을 프리셋으로 옮기는 것은 별도의 의도된 결정이다.

## 휴강 전파

외부 일정 도구가 `schedules.is_canceled`를 직접 뒤집으므로 앱 레이어 훅으로는 감지할 수 없다.
→ DB 트리거가 `booking_events`에 **이벤트만 기록**하고(트랜잭션 안에서 외부 호출 금지), 멱등 처리기가 횟수 복구(예약당 정확히 1회)·기간 연장·알림을 수행한다. 같은 이벤트를 두 번 처리해도 두 번 복구하지 않는다.
